# Is This Topic Big Enough?

> Before starting, decide which steps you actually need.
> Not every topic needs all 6 steps. Using the full system on a small topic wastes time.

```
Ask yourself: "How long would a complete guide on this topic be?"

  One page answer        → Just ask directly. No system needed.
  Example: "What is CSS flexbox?"

  Medium topic (1 week)  → Start at Step 3. Skip Step 1 and Step 2.
  Example: "Learn CSS Grid", "Learn Git basics", "Learn REST APIs"

  Big topic (1+ month)   → Use ALL steps in order.
  Example: "Learn React", "Learn TypeScript", "Learn System Design",
           "Learn Machine Learning", "Learn AWS"

  Huge topic (3+ months) → Use all steps. Break into sub-topics.
                           Run this system once per sub-topic.
  Example: "Become a backend engineer", "Learn full-stack development"
```

**Decision Table:**

| Situation | Steps to Use |
|---|---|
| Quick question or small concept | Just ask directly |
| Know the topic partially, filling gaps | Steps 3 → 4 → 5 → 6 |
| New topic, medium size (1 week) | Steps 3 → 4 → 5 → 6 |
| New topic, big size (1+ month) | All steps 1 → 2 → 3 → 4 → 5 → 6 |
| Already have a guide, need practice | Steps 4 → 5 → 6 |

# Should I Even Learn This?

> Use this BEFORE starting anything.
> Sometimes the topic is not the right next step for you.
> This prompt tells you honestly whether to proceed and why.

---

## Should I Learn This — Prompt

```
I am considering learning [TOPIC].

My current situation:
  - I currently know: [list what you already know]
  - My goal is: [what you want to achieve — job, promotion, project, etc.]
  - My timeline is: [how much time you have]
  - I am considering this because: [why you think you need this topic]

Tell me honestly:

1. Is [TOPIC] the RIGHT next thing for me to learn given my goal?
   If not, what should I learn instead and why?

2. What specific problems does knowing [TOPIC] solve for someone
   at my level with my goal?

3. What are the PREREQUISITES I should know before learning [TOPIC]?
   Do I already have them? What am I missing?

4. What is the MINIMUM viable version of [TOPIC] I need for my goal?
   What can I skip entirely given my specific situation?

5. What is the realistic career or skill impact of learning [TOPIC]
   for someone in my situation?

6. Give me a final verdict:
   LEARN NOW / LEARN LATER / SKIP ENTIRELY
   With a one paragraph explanation of why.
```

# What Do I Already Know?

> Answer these before running Step 1. Your answers change which prompts you use.

```
Answer honestly before starting:

  1. Have I ever used [TOPIC] before?
     YES → Go to Step 3 directly. Add "I already know [X]" to the prompt.
     NO  → Start at Step 1.

  2. Do I know what sub-topics exist in [TOPIC]?
     YES → Go to Step 3. List the scope yourself.
     NO  → Start at Step 1 to discover the scope first.

  3. Am I learning for a specific goal?
     Interview      → Step 4, use Practice Prompt 1
     Senior role    → Step 4, use Practice Prompt 2
     Portfolio      → Step 4, use Practice Prompt 3
     Deep learning  → Step 4, use Practice Prompt 4

  4. How much time do I have?
     Under 2 weeks  → Level 1 only in Step 4
     2 to 4 weeks   → Level 1 + Level 2 in Step 4
     1+ month       → All levels in Step 4

  5. Do I already know a related topic?
     YES → Use the "Connect to What I Know" bonus prompt
           alongside Step 3 to learn faster.
     NO  → Proceed normally.
```

# Step 1 — Blueprint Map

> **Purpose:** Discover everything that exists in the topic before committing to learning it.
> Use this when you know nothing or very little about the topic.
> Skip this step if you already know the scope.

---

## Step 1 Prompt — Copy and paste this

```
I want to learn [TOPIC] using [LANGUAGE / TOOL if applicable].
I am starting from [YOUR CURRENT LEVEL — zero / basics / intermediate].
I do not know what topics exist in this field.

Before teaching me anything, show me:

1. Every topic, sub-topic, and concept that exists in this field
2. The order I should learn them in
3. How long each topic takes (days or weeks)
4. What depends on what — what must I learn before what
5. Grouped into phases or milestones
6. A total timeline estimate at the end

Do NOT teach me anything yet.
Just give me the complete map so I can see the full picture
and understand what I am committing to before I begin.
```

---

## What You Get Back

A complete table of contents of the entire field organized into phases with time estimates. You can now see:

- How big the topic actually is
- What to learn first vs later
- What you can skip for now
- How long it will realistically take

## What to Do After Step 1

```
Is the scope bigger than I expected?
  YES → Break it into sub-topics. Run this system once per sub-topic.
  NO  → Continue to Step 2.

Are there sections I already know?
  YES → Note them. You will skip those in Step 3.
  NO  → Continue to Step 2.
```

# Step 2 — Ecosystem Map

> **Purpose:** Discover what tools, libraries, and technologies exist around your topic.
> Prevents you from learning outdated tools or missing the standard ones.
> Use this for any topic that has a tooling ecosystem.
> Skip this for pure theory topics (algorithms, system design concepts, etc.).

---

## Step 2 Prompt — Copy and paste this

```
I am learning [TOPIC] and I want to understand the complete
ecosystem of tools, libraries, and technologies around it in [CURRENT YEAR].

Tell me:

1. Every major tool, library, and framework that exists in this ecosystem
2. For each tool: what it does, what it is best for, and who uses it
3. What is the current industry standard in [CURRENT YEAR]
4. What is outdated or being replaced and what replaced it
5. What a beginner should use vs what a senior uses
6. The recommended starter stack for someone learning [TOPIC] today

Organize by category.
Include a "Quick Picker" at the end:
  "I need to do [X]" → "Use [TOOL] because [REASON]"

Do NOT teach me how to use any tool yet.
Just show me the complete map of what exists so I can
understand the ecosystem before choosing what to learn.
```

---

## What You Get Back

A complete reference of every tool in the ecosystem with clear guidance on what to actually use. You will know what is worth learning and what to ignore.

## What to Do After Step 2

```
Look at the recommended starter stack.
Note which tools appear in your Step 1 map.
These are what you will focus on in Step 3.
```

# Step 3 — Complete Guide

> **Purpose:** Get a complete teaching document covering everything from your map.
> This is the main learning document you will read and study.
> Generate in parts for big topics to avoid hitting response limits.

---

## Step 3 Prompt — Copy and paste this

```
I need a COMPLETE guide for [TOPIC].

ROLE: This is for someone who [YOUR CURRENT LEVEL].
Example: "knows JavaScript basics but has never used React"
Example: "is an absolute beginner with zero programming knowledge"
Example: "knows Python basics and wants to learn Machine Learning"

GOAL: After reading this guide, I should be able to:
  - [Outcome 1 — what you want to DO]
  - [Outcome 2 — what you want to BUILD]
  - [Outcome 3 — what you want to UNDERSTAND]

SCOPE — Cover ALL of these topics:
  [Paste the phases and topics from your Step 1 map here]
  [Or list the specific topics you want covered]

FORMAT:
  - Single markdown document
  - Clickable table of contents at the top
  - Back to table of contents link after every section
  - All code in [LANGUAGE] with inline comments explaining every line
  - Solved examples with step-by-step trace tables where relevant
  - Comparison tables where relevant
  - "When to use" AND "when NOT to use" for every concept
  - Quick revision one-liners at the end of every section

LEVEL: [Absolute beginner / knows basics / intermediate / advanced]

STYLE: [Simple and friendly / concise / detailed and exhaustive]

GENERATE IN PARTS:
  Generate Part 1 first covering [first 3-4 topics from your scope].
  After I say "Generate Part 2", continue with the next section.
  Do not generate everything at once.
```

---

## Tips for Step 3

```
For big topics: always generate in parts.
  Say "Generate Part 1" → read it → say "Generate Part 2" → and so on.
  This gives you better quality per section.

For medium topics: you can ask for everything at once.
  Just remove the "GENERATE IN PARTS" section from the prompt.

If you already know some sections from Step 1:
  Remove those from the SCOPE list.
  Add "I already know [X], skip that section" to the prompt.

If something in the guide does not click:
  Use the "Explain It Differently" bonus prompt.

If you know a related topic already:
  Use the "Connect to What I Know" bonus prompt alongside this.
```

## Which Practice Prompt?

```
What is your primary goal right now?

  "I have an interview coming up"
    → Use Practice Prompt 1 — Interview Prep

  "I want to think like a senior / architect"
    → Use Practice Prompt 2 — Senior / Architecture

  "I want to build real portfolio projects"
    → Use Practice Prompt 3 — Real Product Builder

  "I want one guide covering everything"
    → Use Practice Prompt 4 — Master (All Combined)

## How to Generate Parts

After copying any practice prompt below, add ONE of these commands at the end:

```
"Generate Part 1 — Level 1, Patterns 1 to 6"
"Generate Part 2 — Level 2, Patterns 7 to 11"
"Generate Part 3 — Level 3, Patterns 12 to 14"
"Generate Part 4 — Level 4, Patterns 15 to 19"
```

Or generate one pattern only:
```
"Generate only Pattern [N] — [Name]. All scenarios. Complete. No placeholders."
```

Regenerate a specific scenario:
```
"Regenerate Pattern [N] Scenario [N]. Make it harder / different company."
```

# Step 5 — How to Actually Do the Practice

> This is the step most people skip. It is also the step that determines whether you actually learn.

---

## The Non-Negotiable Rules

```
Rule 1 — Try it yourself FIRST.
  Before reading the solution, spend at least 20-30 minutes
  actually trying to solve it yourself.
  Struggling is not a sign you are doing it wrong.
  Struggling IS the learning.

Rule 2 — Read to understand, not to copy.
  When you read the solution, understand WHY every line exists.
  If you cannot explain a line, you do not understand it.

Rule 3 — Close it and write it again.
  After reading the solution, close it.
  Write the solution again from memory.
  This is where real skill builds.
  If you cannot write it, go back and read again.

Rule 4 — Take notes on what you got wrong.
  Every mistake is a lesson worth writing down.
  Use the Note Taking Template in this document.
  Review these notes before your next session.

Rule 5 — Do not move on until you can do it without hints.
  Speed is not the goal. Understanding is.
  One pattern done properly > five patterns skimmed quickly.

Rule 6 — Check your level honestly after every 2 patterns.
  Use the "What Level Am I At" bonus prompt to get
  an honest external assessment of where you actually stand.
  Do not rely only on how confident you feel.
```

---

## The Honest Truth About Time

```
If you do the practice properly:
  Time:    Full estimate
  Benefit: 100%

If you read without doing:
  Time:    Half the estimate
  Benefit: 20%

If you just skim solutions:
  Time:    Quarter the estimate
  Benefit: 5%

The rule that matters most:
  30 minutes struggling yourself
  is worth more than
  2 hours reading solutions.
```

# Step 6 — Build a Real Project

> **Purpose:** Consolidate everything you learned into one real project.
> This is what makes the learning permanent and gives you something to show.
> Do this after completing at least Level 1 and Level 2 of practice.

---

## Step 6 Prompt — Copy and paste this

```
I have just finished learning [TOPIC] by studying [what you covered].
I want to build a real project that uses as much of what I
learned as possible and is good enough to show in a portfolio.

Tell me:

1. Give me 3 project ideas at different complexity levels:
   - Beginner project (1-3 days) — tests Level 1 knowledge
   - Intermediate project (1-2 weeks) — tests Level 1 + Level 2
   - Portfolio project (2-4 weeks) — tests everything

2. For the intermediate project, give me:
   - Full feature list (what it should do)
   - Technical requirements (what [TOPIC] concepts it must use)
   - Folder structure / architecture to follow
   - The exact order to build features (what to build first)

   - Day 1: [what to build]

# Note Taking Template

> Copy this for every section you study in Step 3.
> Good notes compound — they save you hours of re-reading later.
> Bad notes are just a dump of information you will never re-read.

---

## How to Take Notes That Actually Help

```
The goal of notes is NOT to record everything.
The goal is to capture:
  1. Things you did not know before
  2. Things that confused you and how you resolved them
  3. The key insight that makes the concept click
  4. The most common mistake to avoid
  5. The one-liner you will use to remember this

If something is easy and obvious to you — do not write it down.
Write only what was NEW or HARD.
```
